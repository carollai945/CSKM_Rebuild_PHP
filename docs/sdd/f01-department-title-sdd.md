# F01 部門與職稱管理 SDD

## 1. 對應需求

- Story：`F01-1`
- 對應 BDD：`bdd/f01-department-title.feature`
- 目標：維護部門與職稱主資料，供人員與權限設定使用

## 2. 角色

- 系統管理員：新增、修改、停用部門/職稱
- 一般員工或主管：不得維護部門/職稱
- 系統：維持主資料完整性並檢查引用關係

## 3. 畫面與路由

- `F01.jsp`：部門/職稱列表
- `F010.jsp` ~ `F012.jsp`：編修頁

### 已知路由

- `GET /F/F01`
- `POST /F/F01/search`
- `POST /F/F01/delete`
- `GET /F/F010`
- `POST /F/F010/new`
- `POST /F/F010/edit`
- `POST /F/F011/new`
- `POST /F/F012/edit`
- `POST /F/F011/getdep`

## 4. API 對應

- `GET /departments`：查詢部門
- `POST /departments`：建立部門
- `PUT /departments/{departmentId}`：修改部門
- `GET /titles`：查詢職稱
- `POST /titles`：建立職稱
- `PUT /titles/{titleId}`：修改職稱

## 5. 資料設計

- 主檔：`department_mgmt`（部門）、`department_mgmt`（職稱層級）
- 參照：`branch_mgmt`（區域）、`hr_management`（人員資料）
- 核心欄位：區域代碼、部門代碼、部門名稱、職稱代碼、職稱名稱、狀態、更新時間
- 部門代碼需全系統唯一，職稱代碼在同部門下不得重複

## 6. 流程設計

1. 管理者查詢部門或職稱資料
2. 進入對應頁面新增或修改部門 / 職稱
3. 系統驗證必填欄位與代碼唯一性
4. 若為停用操作，先檢查是否仍被 `hr_management` 引用
5. 驗證通過後儲存，並回傳最新清單供 F02 引用

## 7. 權限與前端互動

- 僅系統管理者可操作新增、編修、儲存、停用
- 列表需顯示是否停用與編修入口
- 非授權角色不得送出異動請求
- F02 透過部門與職稱主檔資料載入下拉選單

## 8. 錯誤處理

- 重複代碼不得儲存
- 必填欄位缺漏不得儲存
- 已被引用之停用需有提示並拒絕異動
- 無權限者不得修改

## 9. 風險與設計注意事項

- 停用已被引用的部門或職稱會造成 F02 人員資料異常，需強制阻擋
- 部門與職稱主檔異動需即時反映在 F02 下拉選單，避免選單與主檔不一致

## 10. 可測試點

- 部門新增/修改
- 職稱新增/修改
- 代碼唯一性與必填驗證
- 被引用資料不得停用
- 管理權限限制
- 與 F02 的引用一致性
